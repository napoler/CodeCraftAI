# 帮助文档 01：如何使用 Spec (规范)

## 1. 什么是“规范驱动开发” (Spec-Driven Development)？

“规范驱动开发”是我们项目的核心协作流程。它的理念非常简单：**在写任何代码之前，先用清晰的语言描述清楚你要做什么**。

这份“描述”就是一份 **Spec** (Specification，规范)。它是一个Markdown文件，存放在根目录的 `specs/` 文件夹下。

## 2. 为什么我们要用 Spec？

- **达成共识**: 避免开发者做到一半，产品经理或团队成员才发现“啊，我想要的不是这个”。Spec是编码前的“合同”，确保所有人对目标有统一的理解。
- **降低沟通成本**: AI 和人类同事都能通过阅读 Spec 快速了解一个功能的完整背景、目标和实现思路，而不需要反复开会或发送消息。
- **思考更深入**: 编写 Spec 的过程，会强迫你思考各种边界情况、潜在风险和实现细节，从而写出更高质量的代码。
- **让AI更懂你**: 一份好的 Spec 是给 AI 最好的指令。它能让 AI 精准地理解你的意图，生成更符合你期望的代码。

## 3. 如何写一份好的 Spec？

我们提供了一个模板 (`specs/template.md`)，一份好的 Spec 通常包含以下几个部分：

- **背景 (Background)**:
    - 为什么要做这个功能？它解决了什么问题？关联哪个任务 (`tasks/`)？
- **目标 (Goals)**:
    - 这个功能完成后，应该达到什么样的具体效果？目标应该是可以衡量的。
- **非目标 (Non-Goals)**:
    - 明确指出“这次不做什么”，可以有效防止范围蔓延。
- **方案设计 (Proposed Solution)**:
    - 你打算如何实现它？需要修改哪些文件？创建哪些新函数或类？API接口如何设计？
- **风险与问题 (Risks and Questions)**:
    - 实现过程中可能会遇到什么问题？有哪些不确定的地方需要团队讨论？

## 4. 工作流程

1.  **从任务开始**: 在 `tasks/backlog` 中选择一个任务。
2.  **创建 Spec 文件**: 复制 `specs/template.md`，在 `specs/` 目录下创建一个新的 `.md` 文件，文件名与你的功能相关（例如 `user-login-feature.md`）。
3.  **编写 Spec 内容**: 按照模板，详细填写各个部分。
4.  **评审 (Review)**:
    - **AI 评审**: 你可以把 Spec 的内容发给 AI 助手，问它：“你觉得这个设计有没有什么问题？有没有更简单的实现方式？”
    - **团队评审**: 将 Spec 链接发给团队成员，进行讨论，确保大家达成共识。
5.  **开始编码**: 只有当 Spec 通过评审后，才正式开始编码工作。

记住，花在编写 Spec 上的15分钟，可能会为你节省后面数小时的返工时间。
