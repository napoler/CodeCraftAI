# 帮助文档 03：如何记录讨论 (Logs)

## 1. 为什么需要记录讨论？

一个项目成功的关键，不仅仅在于写出好的代码，更在于做出正确的决策。而很多重要的决策，都发生在代码之外的讨论中。

`project_logs/` 目录的存在，就是为了捕捉这些“稍纵即逝”的关键信息，为项目留下宝贵的“记忆”。

## 2. 记录的好处

- **决策可追溯**: 几个月后，当你想知道“我们当初为什么选择用这个方案？”时，日志会给你答案。
- **知识共享**: 新加入的成员可以通过阅读日志，快速了解项目的历史背景和技术选型的原因。
- **AI 协作透明化**: 当你与 AI 协作时，记录下关键的指令和修正过程，可以帮助团队理解 AI 的工作方式，并在出现问题时进行复盘。
- **避免重复讨论**: 将已经形成的结论记录下来，可以防止团队在同一个问题上反复拉扯。

## 3. 我们需要记录什么？

`project_logs/` 目录下主要包含三类日志：

### a. 与 AI 的互动 (`ai_interactions.md`)

- **记录时机**: 当你给 AI 下达复杂指令、AI 犯了错你进行了纠正、或者 AI 帮你解决了大问题时。
- **核心内容**: 记录下你的**指令**、AI 的**反馈**，以及你的**修正**或**结论**。
- **目标**: 把 AI 当作一个真正的团队成员，记录与它的关键协作过程。

### b. 团队内部的讨论 (`team_discussions.md`)

- **记录时机**: 在技术评审会、需求讨论会，或者任何线上/线下的重要讨论之后。
- **核心内容**: 记录**参与者**、**议题**、关键的**观点**和最终的**决策**。
- **目标**: 将口头的、非正式的讨论，转化为书面的、明确的结论。

### c. 复杂问题的调查 (`issue_investigations/`)

- **记录时机**: 当你遇到一个棘手的 Bug 或线上问题，需要花费大量时间去分析和定位时。
- **核心内容**: 在这个目录下为该问题创建一个单独的文件（例如 `fix-database-deadlock-issue.md`）。详细记录你的**分析过程**、**尝试过的解决方案**、**根本原因**以及最终的**修复方案**。
- **目标**: 将问题解决的经验沉淀下来，形成团队的知识库。

## 4. 最佳实践

- **及时记录**: 尽量在讨论结束后或问题解决后马上记录，因为记忆会消退。
- **保持简洁**: 不需要长篇大论，抓住核心要点即可。
- **链接到相关资源**: 如果讨论涉及到一个特定的任务或 Spec，在日志中附上链接。

维护好这份日志，是对你未来自己和团队成员最好的投资。
