# 帮助文档 02：如何管理 Tasks (任务)

## 1. 什么是“项目管理即代码”？

我们采用一种创新的方式来管理项目进度：**将任务本身视为代码一样，用 Git 来进行版本控制和状态追踪**。

这意味着，我们不需要依赖任何外部的项目管理工具（如 Jira, Trello）。我们所有的任务都以 Markdown 文件的形式，存放在根目录的 `tasks/` 文件夹中。

## 2. 为什么这样做？

- **零成本**: 无需购买、配置和学习任何外部工具。
- **透明和开放**: 任何人只要能访问代码仓库，就能看到完整的任务看板和历史。
- **与开发流程无缝集成**: 任务的变更（例如，从“待办”到“进行中”）和代码的提交可以一起进行，关联性极强。
- **AI 友好**: AI 可以很容易地通过文件操作来理解和更新任务状态。

## 3. `tasks/` 目录结构

`tasks/` 目录就像一个看板，它包含几个代表状态的子目录：

- **`backlog/`**: 待办事项。所有新的、未经评估的任务都放在这里。
- **`todo/`**: 计划处理。团队已经评估过，并计划在近期开始的任务。
- **`in_progress/`**: 正在进行。已经有开发者或AI正在处理的任务。
- **`done/`**: 已完成。已经开发、测试并合并到主分支的任务。
- **`archive/`**: 归档。对于已经完成很久，或者被取消的任务，可以移到这里，以保持看板的整洁。

## 4. 工作流程

这个流程的核心就是 **移动文件**。

1.  **创建任务**:
    - 在 `tasks/backlog/` 目录下，创建一个新的 `.md` 文件。
    - 文件名应该简短地描述任务，例如 `add-user-login-button.md`。
    - 文件内容可以更详细地描述任务需求。

2.  **开始任务**:
    - 当你决定开始处理一个任务时，首先要为它创建一个对应的 Spec (详见 `01-how-to-use-specs.md`)。
    - 然后，使用 `git mv` 命令将任务文件从 `todo/` 或 `backlog/` 移动到 `in_progress/` 目录。
    - **示例**:
      ```bash
      git mv tasks/todo/add-user-login-button.md tasks/in_progress/add-user-login-button.md
      ```
    - 将这个文件移动操作，与你的代码提交放在同一个 Pull Request 中。这样，任何人都能从 PR 中看到你正在处理哪个任务。

3.  **完成任务**:
    - 当你的代码被成功合并到 `main` 分支后，最后一步就是将任务文件从 `in_progress/` 移动到 `done/` 目录。
    - **示例**:
      ```bash
      git mv tasks/in_progress/add-user-login-button.md tasks/done/add-user-login-button.md
      ```
    - 这通常是你合并 PR 后的一个快速跟进提交。

通过这种方式，我们的项目状态始终与代码库保持着 **100% 的同步**。
