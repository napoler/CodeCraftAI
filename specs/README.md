# 规范驱动开发 (Spec-Driven Development)

欢迎来到我们的规范文档中心！本文档旨在帮助团队成员理解并遵循我们的开发工作流程。

## 1. 核心理念

我们采用 GitHub 的 **规范驱动开发 (Spec-Driven Development)** 方法论。这意味着在投入实际编码之前，我们必须首先通过编写和评审一份“规范 (Spec)”文档，来确保所有相关方（包括开发、产品、设计等）对“做什么”和“为什么做”达成清晰、全面且书面的共识。

**规范文档是项目的唯一真实来源 (Single Source of Truth)。** 代码实现必须严格遵循已批准的规范。

## 2. 规范的生命周期

每一项有意义的功能开发、技术改进或架构决策都始于一份规范。其生命周期如下：

1.  **草案 (Draft)**
    *   **负责人**: 任何团队成员。
    *   **操作**: 基于 [`template.md`](./template.md) 创建一个新的规范文件，概述你发现的问题、要实现的目标以及初步的解决方案。
    *   **产出**: 一个新的 `.md` 文件，例如 `specs/2023-11-02-new-feature.md`。

2.  **评审 (Review)**
    *   **负责人**: 规范的作者。
    *   **操作**: 提交一份 Pull Request (PR)，将你的规范草案添加到本 `specs` 目录下。在 PR 中 @ 相关同事进行评审。
    *   **目标**: 通过异步或同步讨论，收集反馈，完善方案。评审者应关注目标的清晰度、方案的可行性、潜在风险等。

3.  **批准 (Approved)**
    *   **负责人**: 项目负责人或指定的技术负责人。
    *   **操作**: 当规范经过充分讨论并达成共识后，负责人合并该 PR。
    *   **状态**: 此时，规范被视为团队的“合同”，成为后续开发工作的唯一依据。

4.  **实现 (Implementation)**
    *   **负责人**: 指定的工程师。
    *   **操作**: 根据已批准的规范进行编码工作。在相关的代码提交和 Pull Request 中，必须附上对应规范文档的链接。
    *   **要求**: 代码审查（Code Review）时，评审者也应以规范为准绳，检查实现是否偏离了约定。

5.  **完成 (Final)**
    *   **负责人**: 功能开发者。
    *   **操作**: 在功能成功上线、部署后，如果最终实现与原始规范有细微差别，应回到规范文档中进行更新，以确保其准确反映最终状态。
    *   **目的**: 保持文档的长期价值，使其成为未来理解系统行为的可靠参考。

## 3. 如何开始

1.  **复制模板**: 将 [`template.md`](./template.md) 复制一份，并根据你的功能或议题重命名。
2.  **填写内容**: 尽可能详细地填写模板中的各个部分。
3.  **提交评审**: 创建一个 Pull Request，邀请大家参与讨论。
